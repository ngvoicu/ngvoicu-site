SpecKit — Practical Guide for a Multi-Repo Workspace

by Gabriel Voicu

A step-by-step how-to for using GitHub Spec Kit across a multi-repo workspace, with a single git-tracked products-specs/ as the spec home. The worked example uses a fictional ACME workspace — substitute your own repo names.

Folder pill convention. Every command block is preceded by a location marker telling you where you should be when you run it. Four values:

>

- 📍 anywhere — system-wide commands (install)

- 📍 in parent acme/ — at the workspace root, where all repos live as siblings

- 📍 in products-specs/ — inside the spec-only repo; this is where SpecKit lives

- 📍 in <repo-name>/ — inside one of your code repos (e.g. acme-backend-core/)

1. What this guide is

SpecKit is a CLI (specify) plus a set of slash commands (/speckit.specify, /speckit.plan, /speckit.tasks, /speckit.implement, /speckit.constitution, /speckit.clarify, /speckit.analyze) that turn a feature description into a structured spec, plan, task list, and implementation. Out of the box it assumes one codebase. This guide adapts it to a workspace where specs live in their own repo and code lives in multiple sibling repos.

Two mechanisms do the multi-repo work:

  • Constitution.md — a markdown file that lists your repos and their routing rules. Read by /speckit.plan on every call.
  • Customized templates — small edits to plan-template.md and tasks-template.md so plan.md and tasks.md always carry "Affected Repositories" and per-task [REPO] tags.

Together they make every plan and tasks file know which repos a feature touches.

2. Prerequisites

  • OS: macOS, Linux, or Windows (PowerShell supported).
  • Python ≥ 3.11.
  • uv (recommended) or pipx for installing the CLI.
  • Git.
  • An AI coding agent that supports SpecKit (Claude Code, GitHub Copilot, Cursor, Windsurf, Gemini CLI, Codex, etc.). Pick one to use as <your-agent> below.
  • A workspace with multiple sibling repos. The example uses 6 ACME repos:
~/Projects/acme/
├── acme-backend-core/         # Java 21 + Spring Boot 3 (.git)
├── acme-backend-billing/      # Python 3.12 + FastAPI    (.git)
├── acme-frontend-web/         # Next.js 15 + React 19    (.git)
├── acme-infra/                # Docker + Terraform + OIDC (.git)
├── acme-rest-collections/     # YAML API fixtures        (.git)
└── acme-docs/                 # Markdown                 (.git)

3. Part 1 — One-time setup

Do these seven steps once. After the final commit you have a working SpecKit ready for any feature.

3.1 Install the specify CLI

📍 anywhere

# Pick a real tag from https://github.com/github/spec-kit/releases first
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z

# Verify it's on your $PATH
which specify
specify --version
specify integration list   # lists every supported AI agent

specify is system-wide. You only re-run it when initializing a project, adding a preset, or upgrading. Day-to-day work happens through your agent's slash commands, not the CLI.

3.2 Create the products-specs repo as a sibling

📍 in parent acme/

cd ~/Projects/acme

mkdir products-specs
cd products-specs
git init

# Create the three tier folders SpecKit will read but never write to
mkdir technical-specs functional-specs feature-specs
touch feature-specs/registry.md

3.3 Initialize SpecKit inside products-specs/

📍 in products-specs/

# Replace <your-agent> with one of: claude, copilot, windsurf, gemini, codex, ...
specify init --here --force --integration <your-agent> --no-git --script sh

What got created:

products-specs/
├── .specify/
│   ├── memory/constitution.md    # placeholder — you'll author it next
│   ├── scripts/                  # bash helpers
│   ├── templates/                # spec/plan/tasks templates — you'll customize
│   └── specs/                    # numbered features will land here
├── (your-agent's command folder, e.g. .claude/commands/, .windsurf/workflows/, .github/agents/)
├── feature-specs/                # already exists from Step 3.2
├── functional-specs/             # already exists from Step 3.2
└── technical-specs/              # already exists from Step 3.2

Why --no-git? It tells SpecKit not to create branches when you run /speckit.specify. You don't want SpecKit touching the git history of your sibling repos.

3.4 Author the constitution

📍 in products-specs/

Open .specify/memory/constitution.md in your editor and replace the placeholder with the content below. This file is the single source of truth for cross-repo routing — /speckit.plan reads it on every call.

# ACME Constitution

## Repositories (siblings of products-specs/)

| Repo                  | Stack                                                  | Owns                                          |
|-----------------------|--------------------------------------------------------|-----------------------------------------------|
| acme-backend-core     | Java 21, Spring Boot 3, PostgreSQL 16, Flyway          | Core API: customers, orders, products, search |
| acme-backend-billing  | Python 3.12, FastAPI, PostgreSQL, Alembic              | Billing, invoices, subscriptions, webhooks    |
| acme-frontend-web     | Next.js 15, React 19, TypeScript, TanStack Query       | Customer + admin web app                      |
| acme-infra            | Docker Compose, Terraform, AWS, OIDC provider          | Local + deploy environments, identity config  |
| acme-rest-collections | YAML (OpenCollection)                                  | API request fixtures for both backends        |
| acme-docs             | Markdown                                               | Platform docs                                 |

## Routing rules

A plan MUST identify which repos a feature touches. Heuristics:

- New/changed REST endpoint        -> acme-backend-core OR acme-backend-billing
                                      AND acme-rest-collections
- Billing / invoice / subscription -> acme-backend-billing (and rest-collections
                                      if API surface changes)
- New/changed UI route or component -> acme-frontend-web
- Auth / identity changes          -> acme-infra (OIDC config) AND
                                      both backends
- Documentation only               -> acme-docs

## Source documents (per repo)

When planning a feature, the plan MUST cite the relevant tier specs as
"Source Documents" in plan.md, and tasks MUST link back to them:

| Repo                  | Technical spec                              | Functional spec                              |
|-----------------------|---------------------------------------------|----------------------------------------------|
| acme-backend-core     | ../technical-specs/acme-backend-core.md     | ../functional-specs/acme-backend-core.md     |
| acme-backend-billing  | ../technical-specs/acme-backend-billing.md  | ../functional-specs/acme-backend-billing.md  |
| acme-frontend-web     | ../technical-specs/acme-frontend-web.md     | ../functional-specs/acme-frontend-web.md     |
| acme-infra            | ../technical-specs/acme-infra.md            | ../functional-specs/acme-infra.md            |
| acme-rest-collections | ../technical-specs/acme-rest-collections.md | (none -- pure contract)                      |

## Conventions (cross-repo)

- TIMESTAMPTZ everywhere; snake_case in SQL; explicit constraint names.
- Java side: Spring Data JPA; Flyway migrations.
- Python side: SQLAlchemy 2.x async; Alembic migrations.
- OIDC for auth; never roll your own.
- Validate at system boundaries; trust internal code.
- Tests are not optional for new features.

## Git policy for SpecKit

- SpecKit MUST NOT create git branches in any sibling repo.
- products-specs/ uses its own normal git workflow.
- Each sibling repo uses its own normal feature-branch workflow,
  decoupled from spec IDs.

## File-system boundary (read vs. write)

SpecKit slash commands have these write permissions inside products-specs/:

ALLOWED to write:
  - .specify/                           (templates, scripts, memory)
  - .specify/memory/constitution.md     (via /speckit.constitution only)
  - .specify/templates/                 (manual edits for template overrides)
  - specs/NNN-feature/                  (spec.md, plan.md, tasks.md, contracts/)

READ-ONLY for SpecKit (NEVER write to these):
  - technical-specs/<repo>.md           (hand-curated long-form architecture)
  - functional-specs/<repo>.md          (hand-curated long-form behavior)
  - feature-specs/<slug>/SPEC.md        (curated archive of shipped features)
  - feature-specs/registry.md           (manual index of shipped features)

If a /speckit.* command appears about to write outside its allowed scope
(e.g. a task that says "update technical-specs/acme-backend-core.md"),
STOP and ask the human. Updates to the read-only folders happen OUTSIDE
the SpecKit pipeline -- by hand or in a normal non-/speckit conversation.

## Code-reading policy

During /speckit.plan and /speckit.tasks generation, the agent MAY read
source files in sibling repositories under ../<repo-name>/ for context:
verifying that technical-specs/<repo>.md is current, identifying
existing patterns to reuse, locating insertion points.

Constraints:
  - READ-ONLY during planning. No edits to sibling-repo files.
  - Prefer reading the technical-spec first; reach for actual code only
    when the spec lacks the level of detail needed.
  - When code is read, list paths in plan.md under "Source Code Inspected"
    so a human reviewer can audit which files informed the plan.

WRITES to sibling-repo source files happen only during /speckit.implement,
on tasks that explicitly carry [REPO] and path: per the customized
tasks-template.

## Prior art (use the feature-specs archive)

When generating a plan, the agent SHOULD scan feature-specs/registry.md
for past features whose tags overlap with the current feature, and cite
relevant matches in plan.md under "Prior Art", linking to
feature-specs/<slug>/SPEC.md.

This is read-only access -- the agent MUST NOT modify any file under
feature-specs/.

## Spec taxonomy

- specs/NNN-name/                    <-- SpecKit OWNS (read+write)
- .specify/memory/constitution.md    <-- SpecKit reads; /speckit.constitution updates
- feature-specs/<slug>/SPEC.md       <-- READ-ONLY for SpecKit; manual archive
- functional-specs/<repo>.md         <-- READ-ONLY for SpecKit; manual maintenance
- technical-specs/<repo>.md          <-- READ-ONLY for SpecKit; manual maintenance

Pilot the constitution on the first feature. The pattern of using the constitution for cross-repo routing is not officially documented by GitHub — it works because /speckit.plan reliably reads constitution.md, and the agent follows natural-language rules. If you find the agent dropping the routing on real features, the template overrides in §3.5 are the durable backup.

3.5 Customize the plan template

📍 in products-specs/

Open .specify/templates/plan-template.md. Insert these three sections after the existing "Summary" block and before "Technical Context":

## Affected Repositories
<!-- REQUIRED. The agent MUST fill this from:
       1) the constitution.md "Routing rules" section
       2) any explicit affected-repo list in the spec.md $ARGUMENTS body
     For each repo touched, output a row. If unsure, ask the user before
     proceeding to Phase 0 research. -->

| Repo | Why this feature touches it | Type of change |
|------|------------------------------|----------------|
| [repo-name] | [1-2 sentences] | endpoint / UI / contract / infra / docs |

## Source Documents
<!-- REQUIRED. For each repo listed in "Affected Repositories" above,
     cite the relevant tier docs from products-specs/. The agent MUST
     READ these files (they are read-only inputs -- never edit them) and
     incorporate their content into the Technical Context section below. -->

- **[repo-name]**
  - technical: ../technical-specs/[repo-name].md §[anchor]
  - functional: ../functional-specs/[repo-name].md §[anchor]
  - prior art (optional): ../feature-specs/[slug]/SPEC.md

## Cross-Repo Coordination
<!-- OPTIONAL. Fill only if the feature touches 2+ repos. Document
     the contract surface between them: API shape, event payloads,
     shared types, deployment ordering. -->

- API contract: [GET /api/...] defined in [repo-A], consumed by [repo-B],
  YAML pinned in acme-rest-collections.
- Deployment ordering: [repo-A] -> rest-collections -> [repo-B].

## Source Code Inspected
<!-- OPTIONAL but recommended for non-trivial features. List the
     sibling-repo source files the agent read while drafting this plan.
     Format: relative path from products-specs/, with a one-line note. -->

- ../[repo-name]/path/to/file.ext
  -- [why this file informed the plan]

3.6 Customize the tasks template

📍 in products-specs/

Open .specify/templates/tasks-template.md. Replace the existing "Format Guidelines" block with:

## Format Guidelines

Every implementation task MUST follow this format:

  [ID] [P?] [Story] [REPO] Description
       refs: ../technical-specs/<repo>.md §<anchor>
       path: <path inside the target repo, e.g. src/main/java/com/acme/...>

Where:
  [ID]    = task identifier (T001, T002, ...)
  [P]     = parallel-safe marker (omit if sequential)
  [Story] = user story label (US1, US2, US3) -- from spec.md
  [REPO]  = MANDATORY. One of the repos in constitution.md routing table.
            Tasks without [REPO] are rejected.
  refs:   = MANDATORY for tasks that implement architecture -- cite the
            technical-spec section the task implements.
  path:   = MANDATORY for file-edit tasks. Path is RELATIVE to the target
            repo's root, NOT to products-specs/.

Example:
  T012 [P] [US1] [acme-backend-core] Add OrderExportService.exportToCsv()
       refs: ../technical-specs/acme-backend-core.md §OrderExportService
       path: src/main/java/com/acme/order/OrderExportService.java

Watch out: running specify init --force again later (e.g. to upgrade or add an integration) will overwrite your customized templates with fresh upstream copies. Treat the customized templates as code — commit them before ever re-running specify init --force, and re-apply your edits after init.

3.7 Seed the technical-specs and functional-specs (one file per repo)

📍 in products-specs/

These are hand-curated long-form documents. SpecKit reads them but never writes them. One technical-spec and one functional-spec per repo. Skeleton for one repo — repeat for the others.

technical-specs/acme-backend-core.md:

# acme-backend-core -- Technical Specification

## Stack
Java 21, Spring Boot 3, PostgreSQL 16, Flyway, Maven.

## Architecture anchors
- Controllers -> Services -> Spring Data JPA repositories
- DTOs separate from JPA entities
- OIDC via the identity provider configured in acme-infra

## Key services
| Service              | Responsibility                                |
|----------------------|-----------------------------------------------|
| OrderService         | Order lifecycle: create, ship, cancel         |
| CustomerService      | Customer CRUD + profile                       |
| ProductService       | Catalog and inventory                         |
| OrderExportService   | CSV / Excel export of orders                  |
| SearchService        | Cross-entity search                           |

## Data-model anchors
customers, orders, order_items, products, addresses

## API surface
GET/POST /api/customers, /api/orders/*, /api/products, /api/orders/export

functional-specs/acme-backend-core.md:

# acme-backend-core -- Functional Specification

## Customer-facing behaviors
- Browse products, place orders, track shipment status

## Admin-facing behaviors
### Order management
- Search orders by customer, status, date range
- Trigger CSV export of filtered orders
- Cancel, refund, adjust orders

Repeat for acme-backend-billing, acme-frontend-web, acme-infra, acme-rest-collections, acme-docs.

3.8 Commit the baseline

📍 in products-specs/

git add .
git commit -m "Initial SpecKit + products-specs baseline"

Setup done. You now have: a constitution that knows your six repos, customized templates that force every plan and tasks file to carry multi-repo info, and tier docs that describe each repo's architecture and behavior. The next section is the loop you'll run for every feature.

4. Part 2 — Daily use: shipping a multi-repo feature

Worked example for the rest of this section: "Add a customer-order CSV export, triggerable from the admin orders page." It touches three repos:

  • acme-backend-core — new GET /api/orders/export endpoint, CSV streaming.
  • acme-frontend-web — new "Export CSV" button on the admin orders page.
  • acme-rest-collections — OpenCollection YAML for the new endpoint.

What SpecKit can and can't see.

>

- Always reads: .specify/memory/constitution.md, the templates in .specify/templates/, files in specs/NNN-feature/.

- Reads when you cite the path: technical-specs/<repo>.md, functional-specs/<repo>.md, feature-specs/<slug>/SPEC.md, sibling-repo source files.

- Never auto-discovers anything. No filesystem walking. The agent only reads what's named in the constitution, the spec, or your prompt.

Step 1 — Open the parent in your editor

📍 in parent acme/

<your-editor> ~/Projects/acme

That's it. From here on, your editor and your agent both work from the parent acme/ as the workspace root. This gives you:

  • Slash commands work. Modern agents (Windsurf Cascade, Claude Code, Cursor, Copilot Chat) scan subfolders for command bindings, so the /speckit.* commands installed at products-specs/<agent-folder>/ are discovered automatically.
  • All six sibling repos are visible in the file tree, so you can browse them and the agent can read their source code during planning.
  • One workspace, one mental model. No multi-root setup, no separate CLI launch from a subfolder. Open the parent and start working.

Quick test: in your agent's chat, type /speckit. and watch for autocomplete. You should see specify, plan, tasks, implement, clarify, analyze, constitution. If nothing appears, your agent doesn't auto-discover commands in subfolders — reload the editor window, or check the agent's docs for how to point it at products-specs/<agent-folder>/.

For the rest of this guide:

  • 📍 in parent acme/ on a slash command = type the command in your agent's chat (your editor is rooted there from Step 1).
  • 📍 in products-specs/ or 📍 in <repo-name>/ on a shell command = cd there in your terminal before running it.

Step 2 — Run /speckit.specify

📍 in parent acme/ (in your agent's chat)

In your agent's chat panel, type:

/speckit.specify Add a customer-order CSV export, triggerable from the admin
orders page. Filters from the page (status, date range, customer) pass through.
Affected repos: acme-backend-core, acme-frontend-web, acme-rest-collections.
Cite as Source Documents:
  - products-specs/technical-specs/acme-backend-core.md  §OrderExportService
  - products-specs/functional-specs/acme-backend-core.md §Order management
  - products-specs/technical-specs/acme-frontend-web.md  §Routing

Result: specs/002-order-csv-export/spec.md with the citations embedded in its body. The number (002) auto-increments; first feature was 001.

Step 3 — Run /speckit.clarify if anything is ambiguous

📍 in parent acme/ (in your agent's chat)

/speckit.clarify

The agent asks up to 3 targeted questions. Your answers get appended to spec.md. Skip if the spec is already crisp.

Step 4 — Run /speckit.plan

📍 in parent acme/ (in your agent's chat)

/speckit.plan

Reads spec.md + constitution.md + your customized plan-template.md. Because the template (Step 3.5) requires "Affected Repositories", "Source Documents", and "Cross-Repo Coordination" sections, plan.md is structurally guaranteed to contain them. Expected content:

  • Affected Repositories table — 3 rows (backend-core, frontend-web, rest-collections), each with a "Type of change".
  • Source Documents — per-repo bullets pointing at the cited tier docs.
  • Cross-Repo Coordination — the API contract surface (GET /api/orders/export), defined in backend-core, consumed by frontend-web, pinned in rest-collections.
  • Source Code Inspected — if the agent opened any sibling-repo source files for context.
  • Technical Context, Constitution Check, Phase 0 research, etc. — standard SpecKit sections.

Step 5 — Run /speckit.tasks

📍 in parent acme/ (in your agent's chat)

/speckit.tasks

Generates tasks.md. Because of the customized tasks template, every implementation task carries a [REPO] tag, a refs: citation, and a path: relative to the target repo. Expected output:

T001 [P] [US1] [acme-backend-core] Add OrderExportRequest DTO
       refs: ../technical-specs/acme-backend-core.md §OrderExportService
       path: src/main/java/com/acme/order/export/OrderExportRequest.java

T002 [P] [US1] [acme-backend-core] Add OrderExportService.exportToCsv()
       refs: ../technical-specs/acme-backend-core.md §OrderExportService
       path: src/main/java/com/acme/order/export/OrderExportService.java

T003     [US1] [acme-backend-core] Add GET /api/orders/export controller
       refs: ../technical-specs/acme-backend-core.md §API surface
       path: src/main/java/com/acme/order/export/OrderExportController.java

T004 [P] [US1] [acme-rest-collections] Add OpenCollection YAML
       path: orders/export.yml

T005 [P] [US1] [acme-frontend-web] Add useOrderExport hook
       refs: ../technical-specs/acme-frontend-web.md §Data hooks
       path: src/hooks/useOrderExport.ts

T006     [US1] [acme-frontend-web] Add "Export CSV" button to /orders page
       path: src/app/(admin)/orders/OrderListToolbar.tsx

T007     [POL] [acme-backend-core] Integration test for /api/orders/export
       path: src/test/java/com/acme/order/export/OrderExportControllerIT.java

T008     [POL] [acme-frontend-web] Smoke test the export button
       path: tests/smoke/order-export.spec.ts

Step 6 — Sanity-check tasks.md by hand

📍 in parent acme/ (open products-specs/specs/002-order-csv-export/tasks.md in your editor)

The structural checks (every task has [REPO] and path:) are guaranteed by the template. The remaining human checks are semantic:

  • Are the right repos covered — nothing missed?
  • Are dependencies correctly ordered (backend endpoint before frontend hook)?
  • Do refs: citations point at real anchors in the technical specs (no hallucinated section names)?
  • Are path: values realistic for each target repo's actual layout?

If anything is off, edit tasks.md by hand or re-run /speckit.plan with a corrective hint.

Step 7 — Create per-repo feature branches manually

SpecKit is on --no-git, so it never creates branches. Create them yourself before /speckit.implement:

📍 in parent acme/

cd ~/Projects/acme/acme-backend-core      && git checkout -b feature/order-csv-export   # in acme-backend-core/
cd ~/Projects/acme/acme-frontend-web      && git checkout -b feature/order-csv-export   # in acme-frontend-web/
cd ~/Projects/acme/acme-rest-collections  && git checkout -b feature/order-csv-export   # in acme-rest-collections/
cd ~/Projects/acme/products-specs         && git checkout -b feature/order-csv-export   # in products-specs/

Step 8 — Run /speckit.implement (with babysitting)

📍 in parent acme/ (in your agent's chat)

/speckit.implement

Because every task carries [REPO] and path:, the agent has unambiguous routing instructions. Still watch:

  • Working directory. The agent should cd into the target repo before editing. If it tries to write a path under products-specs/, interrupt — that violates the constitution's file-system boundary.
  • Per-task validation. After every 1–2 tasks, verify the diff is in the right repo:

📍 in parent acme/

  cd ~/Projects/acme/acme-backend-core      && git status   # in acme-backend-core/
  cd ~/Projects/acme/acme-frontend-web      && git status   # in acme-frontend-web/
  cd ~/Projects/acme/acme-rest-collections  && git status   # in acme-rest-collections/
  • Boundary check. No task should propose editing technical-specs/, functional-specs/, or feature-specs/. Those are read-only.

Step 9 — Test per repo

📍 in <repo-name>/

cd ~/Projects/acme/acme-backend-core      && mvn test                              # in acme-backend-core/
cd ~/Projects/acme/acme-frontend-web      && npm run test:smoke && npm run lint    # in acme-frontend-web/
# acme-rest-collections: visual review of the YAML diff

Step 10 — Open per-repo PRs

📍 in <repo-name>/ (one PR per affected repo)

cd ~/Projects/acme/acme-backend-core      && gh pr create --fill   # in acme-backend-core/
cd ~/Projects/acme/acme-frontend-web      && gh pr create --fill   # in acme-frontend-web/
cd ~/Projects/acme/acme-rest-collections  && gh pr create --fill   # in acme-rest-collections/

Each sibling repo uses its own normal feature-branch + PR flow. The shared spec ID (002-order-csv-export) is a convention you cite in PR descriptions, not something SpecKit enforces.

Step 11 — Archive into feature-specs/ — outside the SpecKit pipeline

📍 in products-specs/

**Don't run a /speckit.* command for this.** The feature-specs/ folder is read-only for SpecKit. Do this by hand (or in a normal non-/speckit conversation with your agent).

mkdir -p feature-specs/order-csv-export
cp specs/002-order-csv-export/spec.md \
   feature-specs/order-csv-export/SPEC.md
# Edit feature-specs/registry.md by hand to add the new row.

Step 12 — Update technical-specs/ if architecture shifted — outside the SpecKit pipeline

📍 in parent acme/ (open the file in your editor)

Same boundary. If the architecture genuinely shifted (new service, new data-model entity, new endpoint family), update technical-specs/<repo>.md by hand or via a normal conversation — never as a SpecKit task.

Example: add OrderExportService to technical-specs/acme-backend-core.md §Key services and GET /api/orders/export to the API surface. This is the long-term memory — if it drifts, future plans cite stale documents.

Open products-specs/technical-specs/acme-backend-core.md in your editor and add the new bits.

Step 13 — Commit products-specs/

📍 in products-specs/

git add specs/002-order-csv-export \
        feature-specs/order-csv-export \
        feature-specs/registry.md \
        technical-specs/acme-backend-core.md
git commit -m "feat: order CSV export spec + archive + tech-spec update"

Feature shipped. Repeat Steps 1–13 for the next one. Setup is done forever.

5. The artifact chain

R = SpecKit reads. W = SpecKit writes. Manual = you (or your agent in a normal conversation) author it; SpecKit never writes.

products-specs/
│
├── .specify/memory/constitution.md ──────►  R + W via /speckit.constitution only
│   (routing rules, repo table)              read by /speckit.plan every call
│
├── .specify/templates/plan-template.md ──►  read by /speckit.plan
│   .specify/templates/tasks-template.md ─►  read by /speckit.tasks
│   (your customizations live here)
│
├── technical-specs/<repo>.md ────────────►  R only (Manual)
├── functional-specs/<repo>.md ───────────►  R only (Manual)
├── feature-specs/<slug>/SPEC.md ─────────►  R only (Manual archive)
├── feature-specs/registry.md ────────────►  R only (Manual)
│
│   /speckit.specify                         ▼

6. The 5 disciplines

  1. **Open the parent acme/ in your editor and invoke /speckit.* from your agent's chat there.** The agent discovers the slash commands inside products-specs/<agent-folder>/ automatically. Never cd into a sibling repo to invoke slash commands — the artifact chain only stays coherent if every /speckit.* call shares the same workspace root.
  2. Always include source citations in your /speckit.specify prompt. Even with the customized templates, hand-feeding the citations at specify-time guarantees they're in spec.md from the start.
  3. Verify the Affected Repositories list in plan.md before /speckit.tasks. Catching a missed repo here is much cheaper than catching it after implement.
  4. Always verify per-task git state in each sibling repo. SpecKit doesn't track which sibling repo is on which branch.
  5. Update technical-specs/<repo>.md manually when architecture shifts. By hand, never as a SpecKit task. They are the long-term memory; stale tier docs poison future plans.
